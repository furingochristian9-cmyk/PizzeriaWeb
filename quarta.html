Metodologia di lavoro

Introduzione

Il progetto analizzato descrive il processo completo di sviluppo collaborativo di un sito web per una pizzeria, utilizzando Git e GitHub come strumenti di controllo di versione e di cooperazione tra più sviluppatori. Il lavoro mostra come organizzare un gruppo, gestire un repository remoto, sincronizzare il codice e controllare le modifiche in modo strutturato e professionale.

Creazione del Repository da remoto

Il processo inizia con il Project Manager (PM), che crea un nuovo repository pubblico su GitHub tramite l’opzione New. Viene aggiunto un file README.md, che viene poi modificato per includere le istruzioni del progetto: sviluppatore A, sviluppatore B, sviluppatore C. Il PM invita quindi i membri del gruppo, assegnando loro l’accesso al repository attraverso la sezione Settings > Collaborators.

Inizializzazione

Il PM crea poi una cartella locale e la “inizializza” come repository Git con l’interfaccia Git GUI. Imposta le proprie credenziali e collega il repository locale a quello remoto tramite git remote add. Successivamente effettua il primo allineamento scaricando il file README.md con git pull origin main. Dopo aver aggiunto i file del template del sito web (in particolare index.html), il PM esegue: git status, add, commit git push per inviare tutto sul repository remoto Questo rappresenta la prima versione ufficiale del progetto.

Configurazione degli svilupparoti

Ogni sviluppatore crea la propria cartella locale e la inizializza con git init. Successivamente collega il proprio repository locale al repository remoto tramite: git remote add origin git pull origin main In questo modo ottiene una copia aggiornata del lavoro del PM.

Creazione dei Branch di Lavoro

Vista la seconda regola del GitHub Flow, ogni sviluppatore deve lavorare su un branch personale e descrittivo: menuPizza menuDolci menuBibite Il branch isola il lavoro e impedisce conflitti con il codice principale.

Sviluppo delle funzionalità

Ogni sviluppatore aggiunge un elemento nel proprio rispettivo menù modificando index.html e inserendo eventuali immagini nella cartella /img. Le operazioni principali sono: git add -A git commit -m "Aggiunto item menù" git push origin (nome-branch) Gli altri sviluppatori compiono la stessa procedura nei propri branch.

Pull Request e Revisione del PM

La terza regola del GitHub Flow richiede che ogni modifica venga “pushata” continuamente sul proprio branch. Per integrare il lavoro nel main, gli sviluppatori aprono una pull request. Il PM:analizza le pull request controlla le modifiche apportate può approvare o richiedere rettifiche può inserire commenti direttamente sui file Se necessario, lo sviluppatore aggiorna il branch ed effettua un ulteriore push.

Gestione dei conflitti

In caso di modifiche sovrapposte, GitHub segnala dei conflitti. Il PM li risolve manualmente confrontando le versioni. Una volta corretto il file, procede con: Mark as resolved Merge pull request Il lavoro viene così integrato nella versione pubblicata.

Aggiornamento dei Repository Locali

Dopo ogni merge pull request, gli sviluppatori devono aggiornare le loro copie tramite: git pull origin main in modo da poter continuare a lavorare su una versione coerente del progetto.

Conclusioni

La trattazione dimostra come Git e GitHub siano strumenti essenziali nello sviluppo moderno: permettono di controllare l’evoluzione del progetto, tracciare ogni modifica, collaborare in sicurezza e gestire conflitti in modo efficace. Il sistema di versionamento semplifica lo sviluppo condiviso, garantendo ordine, trasparenza e professionalità.
